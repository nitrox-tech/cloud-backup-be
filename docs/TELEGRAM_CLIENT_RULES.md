# Telegram client rules — specification for frontend & AI integration

This document describes **server-driven rules** for how the client integrates with **Telegram for files**.

**Product policy (schema_version ≥ 3):** the client must **not** use the **Telegram Bot API** for uploading or downloading files (ToS / abuse-policy risk). Instead:

- **Private archive:** use the **logged-in user’s Telegram account** (user session — e.g. MTProto / official patterns) and a **personal** context such as Saved Messages / self chat.
- **Shareable archive:** use a **Telegram supergroup** where **every collaborator is a real group member**. Files are **normal group messages**; Telegram applies normal user/group rules. The app should **invite** people into that group (invite link) so they are not relying on a bot for storage.

The backend still stores **no file blobs**; it only returns **metadata and these rules**.

**Auth (`POST /auth/telegram`):** there is **no** Telegram Login Widget hash verification and **no** server-side bot token. The body must include at least `id` (Telegram user id, number); other fields are optional. The server **trusts** the client — suitable only if you accept impersonation risk or run a closed client.

---

## 1. When to load rules

| Scenario | Action |
|----------|--------|
| User opens app and **JWT is still valid** | `GET /config/telegram` with `Authorization: Bearer <access_token>` |
| User opens app and **JWT expired / no session** | `POST /auth/telegram` — response includes **`telegram_client`** (same shape as `GET /config/telegram`) |
| Only **supergroup / archive onboarding** rules | `GET /config/telegram/archive-group` (JWT required) |

---

## 2. HTTP API summary

| Method | Path | Auth | Response |
|--------|------|------|----------|
| `GET` | `/config/telegram` | Bearer JWT | `TelegramClientRulesResponse` |
| `GET` | `/config/telegram/archive-group` | Bearer JWT | `TelegramArchiveGroupConfigResponse` |
| `POST` | `/auth/telegram` | None | `AuthResponse` including `telegram_client` (JSON body: `{ "id": <telegram_user_id>, "username": "..." }`, `hash` / `auth_date` optional and ignored) |

```http
Authorization: Bearer <access_token>
Accept: application/json
```

**API key (required):** every request must include the shared secret in header `X-API-Key` (name overridable via server `app.api.header-name`). Exceptions: `GET /health`, `OPTIONS` (CORS preflight). Same header is required for `POST /auth/telegram` before JWT exists. Set server-side `API_KEY` / `app.api.key` and use the identical value in the client.

**Identifiers:** use **strings** everywhere in the client for ids in JSON and in path/query (`/folders/{id}`, `folder_id`, etc.). Internal user/folder/file ids are string UUIDs generated by the server/DB. **`telegram_user_id`**, **`telegram_chat_id`**, and file **`message_id`** are **opaque strings** in API and database (TEXT / VARCHAR), not fixed-width integers — `POST /auth/telegram` may send `id` as a JSON number; the server stores it as a string.

## 4. `TelegramClientRulesResponse` (schema_version ≥ 4 for updated `policy_note`; ≥ 3 for Bot API removal)

### 4.1 Top-level

| JSON field | Meaning |
|------------|---------|
| `schema_version` | Bump when shape or semantics change. |
| `integration` | Declares **file_storage_model** and **policy_note** (no Bot API for files). |
| `limits` | Soft hint `max_upload_bytes` for UI / chunking (not Bot-specific). |
| `upload` | `recommended_chunk_bytes`, `retry_max`. |
| `upload_routing` | **Where** logical storage lives: private user space vs shared supergroup. |
| `archive_group` | Naming template + onboarding text for **shareable** roots. |
| `login_widget` | Optional `bot_username` for **Login Widget only**. |

### 4.2 `integration`

| Field | Meaning |
|-------|---------|
| `file_storage_model` | e.g. `telegram_user_client` — implement with **user** Telegram session. |
| `policy_note` | Human-readable: avoid Bot API for files; notes that auth does not verify Telegram widget hash. |

### 4.3 `upload_routing`

#### `private_archive`

| Field | Meaning |
|-------|---------|
| `storage_context` | e.g. `self_chat_saved_messages_or_cloud` — implement per your Telegram user client. |
| `description` | Server-provided explanation. |

#### `shared_archive_supergroup`

| Field | Meaning |
|-------|---------|
| `storage_context` | e.g. `telegram_supergroup`. |
| `description` | Everyone uses the **same** supergroup; files are group messages. |
| `members_must_join_telegram_supergroup` | If `true`, UX must ensure users **join the Telegram group** (invite link), not only `folder_members` in your DB. |

**Branching for AI / FE**

```text
IF root folder is private (shareable === false):
  use private_archive.storage_context with user's Telegram session
ELSE:
  use shared_archive_supergroup; ensure Telegram group membership + same supergroup chat_id for all files in tree
```

**`telegram_chat_id` on shareable root (HTTP API):** the server stores one optional `telegram_chat_id` (string, max **200** characters, e.g. `-100…`) on the **shareable root folder** only (`POST /folders` with `shareable: true` and optional `telegram_chat_id`, or `PUT /folders/{id}/telegram-chat` as owner; `DELETE /folders/{id}/telegram-chat` clears it). Subfolders do not carry their own chat id. For every file in that tree, the client resolves download as **root’s `telegram_chat_id` + file’s `message_id`** inside the supergroup.

### 4.4 `archive_group`

Same as before: `group_title` (`must_start_with`, `title_template`, `segment_meaning`, `example`, `client_validation_regex`, `max_title_length`) and `telegram` (`group_type`, `client_onboarding_hint`) — **no** `bot_must_join_group` in v3.

### 4.5 `login_widget`

Optional public `bot_username` (e.g. deep links). **Not** used by the server for auth verification. **Do not** infer file API from this.

## 6. Example full payload (illustrative)

```json
{
  "schema_version": 4,
  "integration": {
    "file_storage_model": "telegram_user_client",
    "policy_note": "Do not use Bot API for file upload/download in the client. POST /auth/telegram does not verify Telegram Login Widget hash — it trusts the client-supplied telegram user id."
  },
  "limits": {
    "max_upload_bytes": 2147483648
  },
  "upload": {
    "recommended_chunk_bytes": 1048576,
    "retry_max": 3
  },
  "upload_routing": {
    "private_archive": {
      "storage_context": "self_chat_saved_messages_or_cloud",
      "description": "Private archive: store files in the user's own Telegram space..."
    },
    "shared_archive_supergroup": {
      "storage_context": "telegram_supergroup",
      "description": "Shareable archive: invite collaborators into the same Telegram supergroup...",
      "members_must_join_telegram_supergroup": true
    }
  },
  "archive_group": {
    "group_title": {
      "must_start_with": "nitro-tech-cloud-",
      "title_template": "nitro-tech-cloud-{segment}-{suffix}",
      "segment_meaning": "numeric_or_short_archive_identifier",
      "example": "nitro-tech-cloud-123-team-alpha",
      "client_validation_regex": "^nitro-tech-cloud-[0-9]+-[a-zA-Z0-9._-]+$",
      "max_title_length": 128
    },
    "telegram": {
      "group_type": "supergroup",
      "client_onboarding_hint": "Create a Telegram supergroup, set the title..."
    }
  },
  "login_widget": {
    "bot_username": "YourBotUsername"
  }
}
```

```typescript
    };

      group_type: string;
      client_onboarding_hint: string;
    };
  };

}
```
